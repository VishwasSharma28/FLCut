FLCUT — LAUNCH / EXPIRY FIX SUMMARY
=====================================
Date: 2026-06-17


ROOT CAUSE
----------
Next.js App Router caches server component pages by default.

When a short link was first visited while its launchAt was still in the
future, Next.js rendered "Not live yet" and stored that HTML in its
route cache. On every subsequent request — including after the launch
time had passed — Next.js served the same cached HTML instead of
re-running the server component. The time check on the server was
never executed again.

The key detail that made this non-obvious:

  The [shortCode] page calls cookies() and headers() to read the
  visitor cookie and user-agent. In Next.js, calling these functions
  normally opts a page out of caching automatically.

  However, the launchAt/expiresAt checks happen BEFORE those calls
  — in the early-return branches. Because the page exits early without
  ever calling cookies() or headers(), Next.js never detected the page
  as dynamic and cached it anyway.


FILES CHANGED
-------------
  src/app/[shortCode]/page.tsx


EXACT FIX
---------
Added one line at the top of the page:

  export const dynamic = "force-dynamic";

This is a Next.js route segment config option. It tells the framework:

  "Never cache this page. Run the server component from scratch on
   every single HTTP request."

With this in place:
  - new Date() is always the real current time
  - The launchAt comparison is always fresh
  - The expiresAt comparison is always fresh
  - No stale HTML is ever served

No other code was changed. No logic, no analytics, no cookies,
no redirect behavior was touched.


EXPECTED BEHAVIOR AFTER FIX
-----------------------------
  Visit link BEFORE launchAt  →  "Not live yet"
  Visit link AFTER  launchAt  →  redirect to original URL  ✓
  Visit link BEFORE expiresAt →  redirect to original URL  ✓
  Visit link AFTER  expiresAt →  "Link expired"


HOW TO VERIFY
-------------
1. Create a short link with launchAt = 2 minutes from now.

2. Visit the short URL immediately.
   You should see: "Not live yet"

3. Wait 2 minutes (or set a launchAt 1 minute in the past by
   editing directly in the database for a quick test).

4. Visit the same short URL again.
   You should now see: the redirect fires to the original URL.

5. Repeat with expiresAt to verify expiry works the same way.

In development (npm run dev) you can also check the terminal output —
each visit to /<shortCode> should log a new server request, not a
cached response.
