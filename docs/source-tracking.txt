FLCut Source Tracking
=====================

HOW SOURCE DETECTION WORKS
---------------------------
When someone clicks a short link, the browser sends a "User-Agent" header
with every request. This is a text string that identifies the software
making the request. For example:

  Mozilla/5.0 (iPhone; CPU iPhone OS 17_0) ... Instagram 310.0...

The redirect page reads this header and runs it through a function called
parseSource(). That function checks for known keywords in order:

  "Instagram"      → Instagram
  "LinkedIn"       → LinkedIn
  "WhatsApp"       → WhatsApp
  "FBAN" or "FBAV" → Facebook
  "Telegram"       → Telegram
  "Edg/"           → Edge
  "Chrome"         → Chrome
  "Safari"         → Safari
  anything else    → Unknown

The result is stored in the source field of the Click record.

The order matters. Chrome's user-agent also contains "Safari", so Chrome
must be checked first. Edge's user-agent also contains "Chrome", so Edge
must be checked first too.


WHY USER-AGENT IS USED
-----------------------
The User-Agent is the only reliable signal available without asking the user
anything. Every browser and app sends it automatically. It's not perfect,
but it's free, requires no extra setup, and works for most common cases.

The referrer header is another option, but it's often empty — browsers and
apps block it for privacy. User-Agent is more consistently present.


WHY WHATSAPP AND INSTAGRAM ATTRIBUTION IS NOT GUARANTEED
---------------------------------------------------------
When someone shares a link in WhatsApp or Instagram, and the recipient
opens it, there are two possible paths:

1. The app opens the link in its in-app browser.
   In this case, the User-Agent includes "Instagram" or "WhatsApp".
   → Source is correctly attributed.

2. The app opens the link in the device's default browser (Chrome, Safari).
   In this case, the User-Agent looks like a normal browser.
   → Source appears as "Chrome" or "Safari", not Instagram or WhatsApp.

Instagram in particular often uses its own in-app browser on iOS/Android.
WhatsApp on some versions also does this. But it's not consistent across
devices, OS versions, or app versions.

This means:
  - If a link goes viral on Instagram, you might see some "Instagram" clicks
    and many "Chrome" or "Safari" clicks. The Instagram ones are the users
    who stayed in the in-app browser.
  - There is no way to tell from the server side which browser a WhatsApp
    link was opened in, if the user chose their default browser.


LIMITATIONS OF THIS APPROACH
------------------------------
1. User-Agent can be faked.
   Anyone can send any User-Agent string they like. A bot can pretend to be
   Chrome. A developer testing an API might use curl. This is a best-effort
   detection, not a guarantee.

2. Attribution is incomplete for shared links.
   As explained above, links shared on social media are often opened in
   a regular browser, so the source looks like "Chrome" even if it came
   from Instagram.

3. New browsers/apps are not covered.
   The pattern list is static. A new browser or app that becomes popular
   will fall through as "Unknown" until the list is updated.

4. Old click records have no source.
   Clicks recorded before this feature was added will show null in the
   source column. The analytics page shows "—" for these.

5. No referrer cross-check.
   A more accurate implementation would also look at the referrer header
   and cross-check it with the user-agent. This is a future improvement.
